# v1.0.0 Release Notes

Date: 2026-06-13

## Product positioning

The Multi-channel Catalog Commerce Assistant is an AI-assisted admin and customer conversation platform for online shops selling through messaging channels. Commerce decisions remain in deterministic backend services; LLM output is used only for structured intent and attribute extraction.

## Supported catalog model

Generic categories, category presets, catalog attribute definitions, product/variant attributes, aliases, generic resolver traces, and unavailable-demand logging. Fashion color/size compatibility remains supported as a compatibility layer.

## Major features

- FastAPI backend with versioned routers, repositories, services, workers, scheduler, health/readiness, Alembic migrations.
- React TypeScript admin panel for shops, catalog, conversations, orders, analytics, simulator, failed jobs, pilot controls, trust layer.
- Channel-provider abstraction for five messaging providers.
- Deterministic commerce engine: draft orders, confirmation, payment controls, inventory reservations, idempotent callbacks.
- Security: production secret validation, encrypted provider tokens, webhook verification, RBAC/shop isolation, redaction, audit logs.
- Operations: failed jobs, simulator, analytics, health/readiness, pilot modes, emergency stop, decision traces, TRL validation.

## Safety constraints

- LLM must never write directly to the database.
- LLM must never decide final `product_variant_id`, stock, price, discount, payment status, shipping status, or final order status.
- Payment links require customer confirmation unless an authorized admin override with reason is recorded and audited.

## Known limitations

- Real-provider sandbox testing was not completed; pilot deployments must validate webhooks and outbound send per provider.
- Scheduler embedding refresh requires a valid `OPENAI_API_KEY`; invalid keys log errors each cycle but do not stop the scheduler.
- Frontend source integration test skips when only the backend container is mounted (no repo-root frontend tree).

## Upgrade and migration notes

1. Back up PostgreSQL before upgrading.
2. Deploy backend 1.0.0 to staging with `APP_ENV=staging`.
3. Run `cd backend && alembic upgrade head`.
4. Verify `/health`, `/ready`, worker, scheduler, webhooks, simulator, failed-job redaction.
5. Promote to production after the checklist below passes.

## Deployment checklist

- [x] `pytest app/tests -q` passes with PostgreSQL (357 passed, 1 skipped in release audit)
- [x] `alembic upgrade head` passes
- [x] Frontend typecheck, lint, test, build pass
- [x] `docker compose config` and `docker compose build` pass
- [x] Docker smoke: `/health`, `/ready`, worker, scheduler start
- [ ] Provider sandbox validation completed or explicitly accepted as mocked-test-only for pilot
- [ ] Production secrets are non-default; provider tokens encrypted/masked

## Rollback notes

- Keep the previous application image until migrations and smoke tests pass.
- If readiness fails after deployment, stop workers first, roll back images, restore DB snapshot if migrations introduced incompatible state.
- Enable emergency stop before rollback if provider delivery behavior is uncertain.
