# v1.0.0 Release Notes

Date: 2026-06-13

## Product positioning

The Multi-channel Catalog Commerce Assistant is an AI-assisted admin and customer conversation platform for online shops selling through messaging channels. The app keeps commerce decisions in deterministic backend services while using LLM output only as structured intent and attribute extraction input.

## Supported catalog model

The release uses a generic catalog model with categories, category presets, catalog attribute definitions, product and variant attributes, aliases, generic resolver traces, and unavailable-demand logging. Fashion color/size compatibility remains supported as a compatibility layer, not as the only catalog model.

## Major features

- FastAPI backend with versioned routers, repositories, services, workers, scheduler, health/readiness endpoints, and Alembic migrations.
- React TypeScript admin panel for shops, products, conversations, orders, analytics, simulator, failed jobs, system health, pilot controls, and trust-layer review.
- Channel-provider abstraction for Instagram, WhatsApp, Telegram, Bale, and Rubika.
- Deterministic commerce/order engine for draft orders, customer confirmation, payment-link controls, inventory reservations, payment callbacks, and order state transitions.
- Security controls for production secret validation, encrypted provider tokens, webhook verification, RBAC/shop isolation, redaction, audit logs, and rate limits.
- Operational controls for failed jobs, simulator safety, analytics, health/readiness, pilot modes, emergency stop, decision traces, replay, and TRL validation.

## Safety constraints

- LLM output must never write directly to the database.
- LLM output must never decide final `product_variant_id`, stock, price, discount, payment status, shipping status, or final order status.
- Backend deterministic services must own all sensitive commerce state changes.
- Payment links require customer confirmation unless an authorized admin override with reason is recorded and audited.

## Known limitations

- Real-provider sandbox testing was not completed in this environment because credentials were unavailable.
- Full PostgreSQL-backed backend and migration verification requires a running PostgreSQL instance.
- Docker smoke verification requires Docker; Docker was unavailable in the current container.

## Upgrade and migration notes

1. Back up PostgreSQL before upgrading.
2. Deploy backend version 1.0.0 with `APP_ENV=staging` first.
3. Run `cd backend && alembic upgrade head` against a clean staging database and an upgraded staging copy.
4. Verify `/health`, `/ready`, worker startup, scheduler startup, provider webhooks, simulator safety, and failed-job redaction.
5. Promote to production only after the full release checklist passes.

## Deployment checklist

- [ ] `cd backend && pytest -q` passes with PostgreSQL available.
- [ ] `cd backend && alembic upgrade head` passes from a clean PostgreSQL database.
- [ ] `cd frontend && npm run typecheck && npm run lint && npm run test && npm run build` passes.
- [ ] `docker compose config`, `docker compose build`, and `docker compose up -d` pass.
- [ ] `/health` and `/ready` pass.
- [ ] Frontend responds.
- [ ] Worker and scheduler remain running.
- [ ] Provider sandbox validation completed or explicitly accepted as mocked-test-only for pilot.
- [ ] Production secrets are non-default and provider tokens remain encrypted/masked.

## Rollback notes

- Keep the previous application image available until backend migrations and smoke tests pass.
- If readiness fails after deployment, stop workers first, roll back backend/frontend images, and restore the previous database snapshot if migrations introduced incompatible state.
- Disable auto-send or enable emergency stop before rollback if provider delivery behavior is uncertain.
